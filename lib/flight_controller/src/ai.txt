what would be a good name for a trait that represents an IMU in rust
The "best" name depends on whether your trait defines the hardware device itself or the capability of providing motion data. Here are a few solid options:
1. The Direct Approach (Noun-based)
If the trait represents the physical object or a driver:
Imu: Simple, standard, and clear.
InertialSensor: More formal and descriptive; good if you want to distinguish it from GPS or other sensors.
MotionSensor: High-level and easy to understand for users who don't care about the "inertial" terminology.
2. The Functional Approach (Adjective-based)
In Rust, traits are often named after what they do (like Read, Write, or Clone): 
Inertial: Short and fits the "is-a" pattern (e.g., impl Inertial for Mpu6050).
SensesMotion: Very explicit about the capability.
ProvidesOrientation: Specific to the data being returned.
3. The "Embedded-HAL" Style
If you are following the patterns found in the embedded-hal ecosystem:
Accelerometer / Gyroscope: Often, it’s better to split an IMU into two traits. A device can then implement both.
ReadInertialData: Focuses on the action of fetching the telemetry.
Recommendation
Go with Imu if you want a concise name for a "catch-all" trait, or split it into Accelerometer and Gyroscope traits if you want to keep your code modular and generic.

I plan on supporting multiple different IMU chips
Since you’re supporting multiple chips, the Provider Pattern is your best bet. It allows you to write generic code that works with any hardware "driver" that fits the trait.
Here is a recommended naming scheme and structure:
1. The "Interface" Trait: Imu
Keep the trait name simple. It represents the idea of an IMU.
Why: When you write a function, it looks clean: fn calibrate<T: Imu>(device: &mut T).
2. The "Concrete" Structs: [ChipName]
Name your structs after the actual hardware (e.g., Mpu6050, Bno055, Lsm6ds3).
Why: Users need to know which specific chip they are initializing (since pins and addresses vary), but once it's initialized, they can treat it as a generic Imu.
3. The "Wrapper" Struct (Optional): ImuDevice
If you want a single, high-level struct that holds any chip, use a wrapper with a generic type.
Naming: struct ImuDevice<T: Imu> { driver: T }
Example Implementation
rust
// The Trait (The Interface)
pub trait Imu {
    fn read_accel(&mut self) -> [f32; 3];
    fn read_gyro(&mut self) -> [f32; 3];
}

// Concrete Struct 1
pub struct Mpu6050 { /* hardware specific fields */ }
impl Imu for Mpu6050 {
    fn read_accel(&mut self) -> [f32; 3] { [0.0, 0.0, 9.8] }
    fn read_gyro(&mut self) -> [f32; 3] { [0.0, 0.0, 0.0] }
}

// Concrete Struct 2
pub struct Bno055 { /* hardware specific fields */ }
impl Imu for Bno055 {
    // ... implementation for BNO055
}
Use code with caution.

Why this works:
No Name Collisions: Imu (trait) and Mpu6050 (struct) are distinct.
Scalability: Adding a new chip just requires a new struct and an impl Imu.
Interchangeability: You can swap an Mpu6050 for a Bno055 in your main logic without changing anything other than the initialization line.

The different chips share some data members, so I'd like a struct to contain these (like a base class in c++), what is the recommended way to do this
In Rust, there is no struct inheritance, so you can't have a "base class." Instead, the idiomatic way to share data is through Composition.
Here is the best way to structure this:
1. The Common Data Struct
Put all your shared fields (calibration offsets, scale factors, latest readings) into a dedicated struct.
Name Idea: ImuState or ImuMetadata.
2. The Hardware Structs
Each specific chip struct "owns" an instance of that common data.
3. The Trait for Behavior
The trait provides the interface, and you can add a method to the trait that gives access to the common state.
Example Pattern
rust
// 1. Shared data members
pub struct ImuState {
    pub accel_offset: [f32; 3],
    pub gyro_offset: [f32; 3],
    pub scale_factor: f32,
}

// 2. The Trait
pub trait Imu {
    // Shared behavior: access the common state
    fn state(&self) -> &ImuState;
    fn state_mut(&mut self) -> &mut ImuState;

    // Unique behavior: how each chip actually talks to hardware
    fn read_raw(&mut self) -> [f32; 6];
}

// 3. Concrete Implementation
pub struct Mpu6050<I2C> {
    i2c: I2C,            // Hardware-specific
    state: ImuState,     // Shared data
}

impl<I2C> Imu for Mpu6050<I2C> {
    fn state(&self) -> &ImuState { &self.state }
    fn state_mut(&mut self) -> &mut ImuState { &mut self.state }

    fn read_raw(&mut self) -> [f32; 6] {
        // Specific I2C logic for MPU6050
        [0.0; 6]
    }
}
Use code with caution.

Why this is better than C++ inheritance:
Explicit over Implicit: You always know exactly where the data lives (self.state).
Default Implementations: You can write default methods in the trait that use state() to perform calculations (like applying offsets) so you don't have to rewrite calibration logic for every chip.
Refined Naming Suggestion:
Trait: Imu
Common Struct: ImuState
Chip Structs: Mpu6050, Lsm6ds3, etc.

I also wish to be able to use either I2C or SPI for each chip
To handle different communication protocols in Rust, the idiomatic approach is Generics. You define your chip struct with a generic type C (for Connection or Comm) and constrain that type using traits from the embedded-hal crate.
Here is the cleanest way to structure this:
1. The Generic Chip Struct
Instead of making separate structs for Mpu6050I2c and Mpu6050Spi, create one struct that is generic over its interface.
Struct: Mpu6050<C>
Trait: Imu (defines the high-level API)
2. The Implementation Pattern
You can then implement your Imu trait for the struct, but keep the hardware-specific communication logic tucked away.
rust
use embedded_hal::blocking::i2c;
use embedded_hal::blocking::spi;

// The common data we discussed earlier
pub struct ImuState {
    pub scale: f32,
}

// The generic chip struct
pub struct Mpu6050<C> {
    pub comm: C,      // This could be I2C or SPI
    pub state: ImuState,
}

// Implementation for I2C
impl<I2C, E> Mpu6050<I2C> 
where 
    I2C: i2c::WriteRead<Error = E> + i2c::Write<Error = E> 
{
    pub fn new_i2c(i2c: I2C) -> Self {
        Self { comm: i2c, state: ImuState { scale: 1.0 } }
    }
}

// Implementation for SPI
impl<SPI, E> Mpu6050<SPI> 
where 
    SPI: spi::Transfer<u8, Error = E> + spi::Write<u8, Error = E> 
{
    pub fn new_spi(spi: SPI) -> Self {
        Self { comm: spi, state: ImuState { scale: 1.0 } }
    }
}

// Finally, implement the high-level Trait for BOTH
impl<C> Imu for Mpu6050<C> {
    fn read_accel(&mut self) -> [f32; 3] {
        // Use self.comm here (you'll need a way to abstract the read)
        [0.0, 0.0, 0.0]
    }
}
Use code with caution.

Could this also be done using an associated type for the Imu trait
Yes, using an associated type is actually the preferred "Pro" way to do this in Rust. It allows the Imu trait to define what a connection must be able to do without caring how (I2C vs SPI) it does it.
In this pattern, you create a internal "Communication" trait that handles the byte-level work, and the Imu trait uses that as an associated type.
The "Associated Interface" Pattern
rust
// 1. Define what any connection (I2C or SPI) must do for THIS chip
pub trait ImuBus {
    type Error;
    fn read_register(&mut self, reg: u8) -> Result<u8, Self::Error>;
    fn read_registers(&mut self, reg: u8, data: &mut [u8]) -> Result<(), Self::Error>;
}

// 2. The Main Trait uses that Bus as an associated type
pub trait Imu {
    type Bus: ImuBus;
    
    fn bus(&mut self) -> &mut Self::Bus;
    fn state(&self) -> &ImuState;

    // Default implementation: works for ANY chip + ANY bus!
    fn read_accel_x(&mut self) -> f32 {
        let mut bytes = [0u8; 2];
        self.bus().read_registers(0x3B, &mut bytes).ok();
        // logic to convert bytes using self.state() scale factors...
        0.0 
    }
}

// 3. Your generic struct
pub struct Mpu6050<B: ImuBus> {
    pub bus: B,
    pub state: ImuState,
}

impl<B: ImuBus> Imu for Mpu6050<B> {
    type Bus = B;
    fn bus(&mut self) -> &mut Self::Bus { &mut self.bus }
    fn state(&self) -> &ImuState { &self.state }
}
Use code with caution.

Why this is the "Golden Standard" for Rust Drivers:
Zero-Cost Abstraction: The compiler resolves the I2C vs SPI calls at compile time. No if/else checks or virtual tables (vtable) required.
Cleaner Trait Bounds: Instead of every function needing where I2C: Write + Read, you just need where T: Imu.
Testability: You can create a MockBus struct that implements ImuBus, allowing you to unit test your calibration and math logic on your PC without real hardware.
Naming Recommendation:
Trait: Imu
Associated Type: type Bus
Connection Trait: ImuBus or RegisterAccess
Shared Data: ImuState

To bridge the gap between the high-level Imu logic and the low-level hardware, you implement ImuBus for a wrapper struct that holds your I2C or SPI handle.
In Rust, we often use a Newtype (a simple wrapper struct) to implement our own traits on types we don't own (like I2c).
1. The Wrapper Structs
These structs "wrap" the embedded-hal handles and translate your ImuBus calls into I2C or SPI transactions.
rust

    pub i2c: I2C,
    pub address: u8,
}

// A wrapper for SPI (if you need it later)
pub struct SpiInterface<SPI, CS> {
    pub spi: SPI,
    pub cs: CS,
}
Use code with caution.

2. Implementing ImuBus for I2C
This is where the "magic" happens. You define how to read/write registers using the I2C protocol.
rust
pub trait ImuBus {
    type Error;
    fn read_register(&mut self, reg: u8) -> Result<u8, Self::Error>;
    fn read_registers(&mut self, reg: u8, data: &mut [u8]) -> Result<(), Self::Error>;
}

impl<I2C, E> ImuBus for I2cInterface<I2C>
where
    I2C: i2c::WriteRead<Error = E> + i2c::Write<Error = E>,
{
    type Error = E;

    fn read_register(&mut self, reg: u8) -> Result<u8, Self::Error> {
        let mut buffer = [0u8; 1];
        self.i2c.write_read(self.address, &[reg], &mut buffer)?;
        Ok(buffer[0])
    }

    fn read_registers(&mut self, reg: u8, data: &mut [u8]) -> Result<(), Self::Error> {
        self.i2c.write_read(self.address, &[reg], data)
    }
}
Use code with caution.

3. Putting it All Together
Now, your Mpu6050 can be created using an I2cInterface.
rust
// The Generic Struct
pub struct Mpu6050<B: ImuBus> {
    bus: B,
    state: ImuState,
}

// A helper to make a new I2C-based MPU6050
impl<I2C> Mpu6050<I2cInterface<I2C>> {
    pub fn new_i2c(i2c: I2C, address: u8) -> Self {
        Self {
            bus: I2cInterface { i2c, address },
            state: ImuState::default(),
        }
    }
}
Use code with caution.

Why this structure is great:
Separation of Concerns: The Mpu6050 logic only knows about read_register. It doesn't care if it's I2C, SPI, or a Mock for testing.
Error Handling: The type Error associated type allows the error to bubble up from the hardware driver (like an I2C NACK) all the way to your main application.
Flexible Addressing: The I2cInterface stores the address, so the rest of your code doesn't have to keep track of it.
Final Naming Check:
Trait: Imu
Interface Trait: ImuBus
Implementations: I2cInterface, SpiInterface
Concrete Chip: Mpu6050<B: ImuBus>

To handle endianness efficiently in Rust, you should use the built-in from_be_bytes (Big Endian) or from_le_bytes (Little Endian) methods on numeric types. These are highly optimized and often compile down to a single CPU instruction.
Since different IMU chips use different byte orders (e.g., MPU6050 is Big Endian, while many Bosch sensors are Little Endian), you should put this logic in your Trait default methods or a Helper struct.
1. The "Endian-Aware" Trait
You can add a method to your Imu trait that handles the conversion automatically based on a chip's specific requirements.
rust
pub trait Imu {
    type Bus: ImuBus;
    
    // Most IMUs use 16-bit integers for raw data
    fn read_raw_u16(&mut self, reg: u8) -> u16 {
        let mut bytes = [0u8; 2];
        self.bus().read_registers(reg, &mut bytes).ok();
        
        // Use the chip's specific endianness
        if self.is_big_endian() {
            u16::from_be_bytes(bytes)
        } else {
            u16::from_le_bytes(bytes)
        }
    }

    fn is_big_endian(&self) -> bool; // Each chip implementation defines this
}
Use code with caution.

2. The "Refined" Approach (Zero-Cost)
A cleaner "Rust-way" is to handle the conversion immediately after the bus read using the byteorder pattern. This keeps your high-level math logic completely separate from the "wire format."
rust
impl<B: ImuBus> Imu for Mpu6050<B> {
    type Bus = B;
    
    fn read_accel_x(&mut self) -> f32 {
        let mut buffer = [0u8; 2];
        self.bus.read_registers(0x3B, &mut buffer).unwrap();
        
        // MPU6050 is Big Endian
        let raw = i16::from_be_bytes(buffer); 
        
        // Convert to Gs using the shared state's scale factor
        (raw as f32) * self.state.accel_scale
    }
}
Use code with caution.

3. Handling Sign Extension
IMUs almost always return signed integers (i16).
Important: Always convert to i16 before converting to f32. If you convert the raw u16 to f32 first, negative acceleration values will appear as massive positive numbers (e.g., -1 becomes 65535).
Summary of Naming for Data Flow:
Raw Bytes: [u8; 2]
Parsed Integer: i16 (using from_be_bytes or from_le_bytes)
Processed Data: f32 (after applying state.scale_factor)

1. Defining the Async Trait
Since Rust 1.75, you can use async fn directly in traits. This allows your IMU driver to "yield" while waiting for hardware, letting other tasks run.

// The Async Bus Interface
pub trait ImuBus {
    type Error;
    // Note the 'async' keyword
    async fn read_registers(&mut self, reg: u8, data: &mut [u8]) -> Result<(), Self::Error>;
}

// The Main Async IMU Trait
pub trait Imu {
    type Bus: ImuBus;
    fn bus(&mut self) -> &mut Self::Bus;

    async fn read_accel(&mut self) -> [f32; 3] {
        let mut buf = [0u8; 6];
        // .await here allows Embassy to switch tasks during the I2C/SPI transfer
        self.bus().read_registers(0x3B, &mut buf).await.ok();
        // ... conversion logic ...
        [0.0, 0.0, 0.0]
    }
}

#[embassy_executor::task]
pub async fn imu_task(mut imu: Mpu6050<I2cInterface<I2c<'static, I2C1, DMA1_CH3, DMA1_CH2>>>) {
    let sender = IMU_CH.sender();
    let mut ticker = embassy_time::Ticker::after(embassy_time::Duration::from_hz(100)); // 100Hz

    loop {
        // Read data using our async Imu trait
        let reading = ImuReading {
            accel: imu.read_accel().await,
            gyro: imu.read_gyro().await,
        };

        // Send to channel (awaits if the channel is full)
        sender.send(reading).await;

        // Wait for next tick
        ticker.next().await;
    }
}

Why this is powerful:
Decoupling: Your main logic doesn't need to know how the IMU is read; it just waits for a Message on the channel.
Backpressure: If your main task gets bogged down and the channel fills up, the imu_task will automatically pause at sender.send().await until there is room, preventing data overflow.
Concurrency: You can easily spawn a third task (like a wireless transmitter) that also takes a receiver() to send the data over Wi-Fi/Bluetooth without touching the sensor code.

I have another task that receives setpoints from a radio connection. I'd like a task that takes the Imu readings and the setpoints from the radio and uses them as inputs to a PID controller
To handle two different data streams (IMU and Radio) in a single control task, you can use embassy_sync::select. This allows your task to "race" multiple receivers and react immediately to whichever one has new data.
Here is how you can structure the control loop:
1. Define the Setpoint Type
rust
#[derive(Debug, Clone, Copy)]
pub struct Setpoint {
    pub target_pitch: f32,
    pub target_roll: f32,
}

static RADIO_CH: Channel<CriticalSectionRawMutex, Setpoint, 2> = Channel::new();
Use code with caution.

2. The PID Control Task
This task listens to both the IMU Channel and the Radio Channel. When a new IMU reading arrives, it runs the PID calculation using the last known setpoint.
rust
use embassy_sync::select::{select, Either};

#[embassy_executor::task]
pub async fn controller_task() {
    let imu_rx = IMU_CH.receiver();
    let radio_rx = RADIO_CH.receiver();
    
    // Initial state
    let mut current_setpoint = Setpoint { target_pitch: 0.0, target_roll: 0.0 };
    let mut pid = MyPidController::new(1.0, 0.1, 0.01); // Kp, Ki, Kd

    loop {
        // Wait for EITHER the IMU or the Radio
        match select(imu_rx.receive(), radio_rx.receive()).await {
            // Case A: New Radio Setpoint arrived
            Either::Second(new_setpoint) => {
                current_setpoint = new_setpoint;
                defmt::info!("New target: {}", current_setpoint.target_pitch);
            }
            
            // Case B: New IMU data arrived (The "Heartbeat" of the PID)
            Either::First(imu_data) => {
                let output = pid.update(
                    current_setpoint.target_pitch, 
                    imu_data.accel[0] // example: pitch axis
                );
                
                // apply_to_motors(output);
            }
        }
    }
}
Use code with caution.

Why this works:
Asynchronous "Select": The select function is the magic. It suspends the task until at least one future completes. If both happen at once, it picks one (usually the first) and the next loop iteration handles the other.
State Management: By keeping current_setpoint as a local variable, your PID loop always has access to the most recent command from the radio, even if the radio only sends updates once per second while the IMU runs at 100Hz.
Timing: Your PID loop is "clocked" by the IMU task. Every time the IMU sends a message, the controller wakes up, ensuring your control output is always based on the freshest sensor data.
Refined Naming Strategy:
Channel names: IMU_CH and SETPOINT_CH.
Task names: imu_worker, radio_worker, and flight_controller.

I'd like the PID controller task to "signal" a motor task when it has new values to output to the motors
To pass the PID output to a motor task without creating a "wait loop," the most efficient tool in Embassy is a Signal.
While a Channel is great for a stream of data (like sensor readings), a Signal is optimized for "latest-value" updates. It acts like a single-slot mailbox: if the PID task sends a new value before the Motor task has finished the old one, the old one is simply overwritten. This is perfect for motor control where you only ever care about the most recent command. 
1. Define the Signal
Use Signal from embassy_sync. It requires a RawMutex (just like the Channel) to be safe across cores.
rust
use embassy_sync::signal::Signal;
use embassy_sync::blocking_mutex::raw::CriticalSectionRawMutex;

// The data we want to send to the motors
#[derive(Clone, Copy)]
pub struct MotorCommand {
    pub throttle: [f32; 4], // e.g., for a quadcopter
}

// Global signal for motor updates
static MOTOR_SIG: Signal<CriticalSectionRawMutex, MotorCommand> = Signal::new();
Use code with caution.

2. Update the Controller Task 
The controller task "signals" the motor task as the final step of its loop.
rust
#[embassy_executor::task]
pub async fn controller_task() {
    let imu_rx = IMU_CH.receiver();
    // ... PID setup ...

    loop {
        let imu_data = imu_rx.receive().await;
        
        // 1. Calculate PID
        let output = pid.update(setpoint, imu_data.accel);
        
        // 2. "Signal" the motor task with new values
        MOTOR_SIG.signal(MotorCommand { 
            throttle: [output, output, output, output] 
        });
    }
}
Use code with caution.

3. The Motor Task
This task stays suspended (consuming 0% CPU) until MOTOR_SIG.wait() is triggered.
rust
#[embassy_executor::task]
pub async fn motor_task(mut pwm_driver: MyPwmDriver) {
    loop {
        // This yields until controller_task calls .signal()
        let cmd = MOTOR_SIG.wait().await;
        
        // Apply the values to the physical hardware (PWM pins)
        pwm_driver.set_throttles(cmd.throttle);
    }
}
Use code with caution.

Why a Signal is better than a Channel here:
Freshness: If the motor task is slightly slower than the PID loop, a Channel would build up a "backlog" of old commands. A Signal ensures the motor task always gets the freshest data immediately upon waking up.
Simplicity: You don't need to define a buffer size. It is literally just "the latest value."
Multi-Core Note:
Since your Controller is on Core 0 and your Motors might be on either core, the Signal handles the cross-core "wake up" automatically using the hardware's internal event system (SEV/WFE on ARM).

In Embassy, you typically use a SimplePwm (on STM32) or Pwm (on RP2040) driver. To keep your motor task clean and hardware-agnostic, you can wrap these in a MyPwmDriver struct. 
1. The Motor Driver Struct
This struct holds the PWM channels. Since each motor is usually on a different channel of a timer, you store the handles to those channels.
rust
use embassy_stm32::timer::simple_pwm::SimplePwm;
// or for RP2040: use embassy_rp::pwm::Pwm;

pub struct MyPwmDriver<'d, T: Instance> {
    pwm: SimplePwm<'d, T>,
}

impl<'d, T: Instance> MyPwmDriver<'d, T> {
    pub fn new(pwm: SimplePwm<'d, T>) -> Self {
        // Enable the timer and channels
        let mut driver = Self { pwm };
        driver.pwm.enable(Channel::Ch1);
        driver.pwm.enable(Channel::Ch2);
        driver.pwm.enable(Channel::Ch3);
        driver.pwm.enable(Channel::Ch4);
        driver
    }

    pub fn set_throttles(&mut self, throttles: [f32; 4]) {
        let max = self.pwm.get_max_duty();
        
        // Convert 0.0..1.0 float to hardware duty cycle units
        self.pwm.set_duty(Channel::Ch1, (throttles[0] * max as f32) as u16);
        self.pwm.set_duty(Channel::Ch2, (throttles[1] * max as f32) as u16);
        self.pwm.set_duty(Channel::Ch3, (throttles[2] * max as f32) as u16);
        self.pwm.set_duty(Channel::Ch4, (throttles[3] * max as f32) as u16);
    }
}
Use code with caution.

2. The Motor Task (Using the Signal)
This task lives on whichever core you prefer (likely Core 0 alongside the controller). It waits for the MOTOR_SIG we defined earlier.
rust
#[embassy_executor::task]
pub async fn motor_task(mut driver: MyPwmDriver<'static, TIM1>) {
    loop {
        // 1. Wait for the PID controller to signal new values
        let cmd = MOTOR_SIG.wait().await;
        
        // 2. Update the hardware immediately
        driver.set_throttles(cmd.throttle);
        
        // 3. Optional: Add a safety timeout 
        // If no signal is received for 500ms, you could kill the motors here.
    }
}
Use code with caution.

3. Initialisation in Main
You set up the pins and the timer, then pass the driver to the task.
rust
#[embassy_executor::main]
async fn main(spawner: Spawner) {
    let p = embassy_stm32::init(Default::default());

    // Configure pins for PWM (specific to your MCU)
    let ch1 = PwmPin::new_ch1(p.PA8, OutputType::PushPull);
    let ch2 = PwmPin::new_ch2(p.PA9, OutputType::PushPull);
    // ... repeat for 4 channels ...

    // Create the Embassy PWM instance (e.g., at 20kHz for motors)
    let pwm = SimplePwm::new(p.TIM1, Some(ch1), Some(ch2), None, None, khz(20), Default::default());
    
    let driver = MyPwmDriver::new(pwm);

    // Spawn the task
    spawner.spawn(motor_task(driver)).unwrap();
}
Use code with caution.

Why this is effective for Dual-Core:
Decoupled Frequency: The imu_task on Core 1 can run at 1kHz, the controller_task on Core 0 can run at 400Hz, and the motor_task updates exactly when a new PID result is ready. Embassy's Signal handles the cross-core wake-up efficiently.
Safety: If the controller_task crashes or hangs, the motor_task will simply stay at MOTOR_SIG.wait().await, preventing the motors from receiving "stale" commands or spinning out of control.

To handle both STM32 and RP2040 in the same codebase, you use Rust's Cargo Features and Conditional Compilation (#[cfg(...)]).
This allows you to swap out the hardware-specific types (like I2c and Pwm) while keeping your high-level logic (the Imu trait and the controller_task) exactly the same.
1. Update Cargo.toml
Define your hardware "targets" as features.
toml
[features]
stm32 = ["embassy-stm32", "stm32-metapac"]
rp2040 = ["embassy-rp", "rp2040-pac"]
Use code with caution.

2. Create a "Hardware Abstraction" Module
Instead of putting chip-specific code in main.rs, create a hw.rs file that aliases the types based on which flag is active.
rust
// hw.rs
#[cfg(feature = "stm32")]
pub use embassy_stm32::timer::simple_pwm::SimplePwm as PwmDriver;
#[cfg(feature = "stm32")]
pub use embassy_stm32::peripherals::TIM1 as PwmInstance;

#[cfg(feature = "rp2040")]
pub use embassy_rp::pwm::Pwm as PwmDriver;
#[cfg(feature = "rp2040")]
pub use embassy_rp::peripherals::PWM_CH0 as PwmInstance;

// Common type alias used by your tasks
pub type MotorPwm = PwmDriver<'static, PwmInstance>;
Use code with caution.

3. Conditional Initialization in main.rs
Your entry point needs to change slightly to call the correct HAL initialization.
rust
#[embassy_executor::main]
async fn main(spawner: Spawner) {
    // 1. Initialize the correct HAL
    #[cfg(feature = "stm32")]
    let p = embassy_stm32::init(Default::default());

    let p = embassy_rp::init(Default::default());

    // 2. Initialize PWM (Hardware specific setup)
    let pwm = setup_pwm(p); 
    
    // 3. The rest of your code is hardware-agnostic!
    let driver = MyPwmDriver::new(pwm);
    spawner.spawn(motor_task(driver)).unwrap();
}

You can hide the messy pin-mapping logic in small functions gated by the same flags:

Cleaner Logic: Your controller_task and imu_task don't have a single cfg flag in them—they just use the MotorPwm type alias.
Faster Iteration: You can even add a feature = "simulator" to run your PID logic on a PC for debugging.

To keep your ESP32 PWM frequency consistent with the 20kHz we set for the STM32 and RP2040, we use the LEDC (LED Control) peripheral. In the ESP32 ecosystem, LEDC is the standard way to generate high-speed PWM for motors or LEDs.
1. The ESP32 PWM Implementation
The ESP32 requires you to define a Timer (which sets the frequency) and then attach Channels to that timer.
rust
#[cfg(feature = "esp32")]
pub fn setup_pwm(p: Peripherals) -> MyPwmDriver {
    use esp_hal::ledc::{Ledc, LSGlobalControl, LowSpeed, Timer, TimerSpeed};

    // 1. Initialize the LEDC peripheral
    let mut ledc = Ledc::new(p.LEDC);

    // 2. Configure the Timer to 20kHz
    // The resolution (e.g., 10-bit) determines the max duty cycle value (0..1023)
    ledc.set_global_config(LSGlobalControl::FreeRunning);
    let mut timer = ledc.get_timer::<LowSpeed>(esp_hal::ledc::timer::Number::Timer0);
    timer.configure(esp_hal::ledc::timer::config::Config {
        duty_resolution: esp_hal::ledc::timer::config::DutyResolution::Bit10,
        clock_source: esp_hal::ledc::timer::config::ClockSource::APB,
        frequency: 20_000.hz(), // 20kHz
    }).unwrap();

    // 3. Create the channels for the 4 motors
    let mut ch1 = ledc.get_channel::<LowSpeed>(esp_hal::ledc::channel::Number::Channel0, p.GPIO0);
    let mut ch2 = ledc.get_channel::<LowSpeed>(esp_hal::ledc::channel::Number::Channel1, p.GPIO1);
    // ... repeat for 4 channels ...

    ch1.configure(esp_hal::ledc::channel::config::Config {
        timer: &timer,
        duty_pct: 0,
        pin_config: esp_hal::ledc::channel::config::PinConfig::PushPull,
    }).unwrap();

    // Return your wrapped driver
    MyPwmDriver { ch1, ch2, max_duty: 1023 }
}

going back to the Imu trait, if I add a function set_acc_offset, is it possible to implement this just once, rather than for every type of Imu
Yes, you can do this using Default Trait Methods. Since all your IMU structs share the ImuState data through the state_mut() method we defined earlier, you can write the logic once in the trait definition.
Every struct that implements Imu will automatically inherit this function.
The Implementation
You define the function body directly inside the trait block:
rust
pub trait Imu {
    type Bus: ImuBus;

    // These must be implemented by each chip (Mpu6050, Bno055, etc.)
    fn state(&self) -> &ImuState;
    fn state_mut(&mut self) -> &mut ImuState;

    // --- SHARED LOGIC (Implemented only once) ---
    fn set_acc_offset(&mut self, offset: [f32; 3]) {
        let s = self.state_mut();
        s.accel_offset = offset;
        log::info!("New accelerometer offset applied.");
    }

    fn apply_calibration(&self, raw: [f32; 3]) -> [f32; 3] {
        let s = self.state();
        [
            raw[0] - s.accel_offset[0],
            raw[1] - s.accel_offset[1],
            raw[2] - s.accel_offset[2],
        ]
    }
}
Use code with caution.

Why this is the "Rust Way":
Code Reuse: You don't have to copy-paste the offset math into every single driver.
Access Control: By requiring the chip to provide state_mut(), you guarantee the trait has a place to store the data without needing C++-style inheritance.
Overrideable: If a specific high-end chip (like the BNO055) has internal hardware registers for offsets, you can "override" this default method in that specific impl block to write directly to the chip's memory instead of your local ImuState.
Pro-Tip: The "Zero-Init" Pattern
To make this even cleaner, ensure your ImuState implements Default. Then, your chip structs can just call ImuState::default() in their new() functions.

is it possible to have some data in the state mutable while some is not
In Rust, mutability is determined by how you hold the reference to the struct, not by the individual fields. However, you can achieve "partial mutability" using two common patterns:
1. The "Split State" Pattern (Recommended)
The cleanest way is to split your data into two structs: one for Configuration (read-only after setup) and one for Status (mutable).
rust
pub struct ImuConfig {
    pub hardware_id: u8,      // Fixed
    pub i2c_address: u8,      // Fixed
}

pub struct ImuStatus {
    pub accel_offset: [f32; 3], // Can change
    pub is_calibrated: bool,    // Can change
}

pub struct Mpu6050 {
    pub config: ImuConfig,      // Stored as-is
    pub status: ImuStatus,
}
Use code with caution.

In your trait, you provide a read-only getter for config and a mutable getter for status. This prevents your calibration logic from accidentally overwriting the hardware address.
2. Internal Mutability (Cell / RefCell)
If you want the state struct to appear "immutable" to the outside world but still allow specific fields to change internally, use Cell.
Cell<T> is perfect for small copyable types like f32 or bool.
It allows you to change the value even if you only have a shared (&self) reference.
rust
use core::cell::Cell;

pub struct ImuState {
    pub hardware_id: u8,               // Normal field (Immutable via &self)
    pub accel_offset_x: Cell<f32>,     // Interior Mutability
}

impl ImuState {
    pub fn update_offset(&self, new_x: f32) {
        self.accel_offset_x.set(new_x); // Works even with &self!
    }
}
Use code with caution.

Which one to choose?
Use Pattern 1 (Split State) if you want the compiler to strictly enforce that "Config" shouldn't be touched during normal operation. This is the most "Rustacean" way. [1]
Use Pattern 2 (Cell) if you are in a constrained environment where you can't easily pass &mut self around (e.g., inside an interrupt or a shared callback). [2]
Pro-Tip: In embedded systems, keep as much as possible in a "Config" struct. You can even put it in Flash memory (via the #[link_section] attribute) to save precious RAM. [3]

3. Safety Warning: Race Conditions
In a multi-core or interrupt-driven environment, a standard RMW cycle is not atomic. If an interrupt or the other core modifies the register between your read and your write, their changes will be lost.

can you give an example implementation of the write_register function
To implement write_register, you add it to your ImuBus trait. For I2C, this usually involves sending a 2-byte payload: the register address followed by the data byte.
1. Update the ImuBus Trait
First, add the method to your trait definition so all interfaces (I2C/SPI) provide it.
rust
pub trait ImuBus {
    type Error;
    async fn read_register(&mut self, reg: u8) -> Result<u8, Self::Error>;
    async fn read_registers(&mut self, reg: u8, data: &mut [u8]) -> Result<(), Self::Error>;
    
    // The new write method
    async fn write_register(&mut self, reg: u8, value: u8) -> Result<(), Self::Error>;
}
Use code with caution.

2. Implementation for I2C (Embassy)
For I2C, you send the register and value together in one write operation.
rust
impl<I2C> ImuBus for I2cInterface<I2C>
where
    I2C: embedded_hal_async::i2c::I2c,
{
    type Error = I2C::Error;

    async fn write_register(&mut self, reg: u8, value: u8) -> Result<(), Self::Error> {
        // We send [Address, Data] as a single transaction
        self.i2c.write(self.address, &[reg, value]).await
    }
    
    // ... read_register etc ...
}
Use code with caution.

3. Implementation for SPI (Embassy)
SPI is slightly different. You usually pull the Chip Select (CS) pin low, send the register address (often with the MSB set to 0 for "write"), send the data, and then pull CS high.
rust
impl<SPI, CS> ImuBus for SpiInterface<SPI, CS>
where
    SPI: embedded_hal_async::spi::SpiDevice,
    CS: embedded_hal::digital::v2::OutputPin,
{
    type Error = SPI::Error;

    async fn write_register(&mut self, reg: u8, value: u8) -> Result<(), Self::Error> {
        // Many IMUs require bit 7 to be 0 for a WRITE operation
        let reg_address = reg & 0x7F; 
        
        // SPI write: [Register, Value]
        self.spi.write(&[reg_address, value]).await
    }
}
Use code with caution.

Why this is the "Embassy Way":
Buffer Safety: We pass a slice &[reg, value]. Embassy handles the DMA transfer (if configured) so the CPU doesn't sit in a busy-loop waiting for the I2C clock.
Error Bubbling: If the slave device doesn't NACK or the bus is noisy, the Result::Err travels all the way back to your modify_pwr_mgmt function so you can handle it.
Pro-Tip: Sequential Writes
If you need to write multiple registers at once (like loading a large configuration table), it's more efficient to add a write_registers(reg, data: &[u8]) method to the trait to avoid the overhead of multiple start/stop conditions on the bus.
